# Lesson 006: IAM Security, Resource Policies, & API Keys

## Objectives

- Understand the differences between Identity-Based and Resource-Based Policies in Bedrock.
- Implement least-privilege IAM policies for the Bedrock Runtime and Converse APIs.
- Set up Service-Linked Roles for model access and training isolation.
- Design cross-account access and boundary-restricted IAM policies.

---

## Theory

Security is a primary pillar of the AWS Well-Architected Framework. In Amazon Bedrock, access control is governed by **AWS Identity and Access Management (IAM)**. There are no proprietary "Bedrock API Keys" as found in other AI provider networks. Instead, all API authorization uses standard AWS IAM SigV4 signatures.

### 1. Key Bedrock API Actions

To configure least-privilege access, you must restrict developer or runtime environments to only the specific API endpoints they need:

- `bedrock:InvokeModel`: Allows standard synchronous model generation (e.g. sending a single query and waiting for the full response).
- `bedrock:InvokeModelWithResponseStream`: Allows streaming responses (e.g. word-by-word generation).
- `bedrock:Converse`: Allows multi-turn structured dialogues using the modern, unified Converse interface.
- `bedrock:ConverseStream`: Allows multi-turn dialogues with streaming outputs.
- `bedrock:ListFoundationModels` & `bedrock:GetFoundationModel`: Allows listing and inspecting active models.

### 2. Resource-Level Restrictions

To enforce strict security, do not use `Resource: "*"` in your production IAM statements. Instead, scope access to specific model ARN formats:

```text
arn:aws:bedrock:<region>::foundation-model/<model-id>
```

For example, to permit access only to Claude 3.5 Sonnet in Northern Virginia:

```text
arn:aws:bedrock:us-east-1::foundation-model/anthropic.claude-3-5-sonnet-20241022-v2:0
```

## AWS Console Walkthrough

### Step 1: Creating a Least-Privilege IAM Policy

1. Open the **AWS Console** and search for **IAM**.
2. Click **Policies** in the left sidebar and click **Create policy**.
3. Under the Service section, search for **Bedrock** and select it.
4. Under Actions, check:
   - Read: `GetFoundationModel`, `ListFoundationModels`
   - Write: `Converse`, `ConverseStream`, `InvokeModel`, `InvokeModelWithResponseStream`
5. Under Resources, select **Specific** and click **Add ARNs** to restrict access to approved model families (e.g. Claude or Nova models).
6. Click **Next**, name the policy `BedrockLeastPrivilegePolicy`, and click **Create policy**.

### Step 2: Associating the Role to a Compute Service

1. In the IAM sidebar, click **Roles** and click **Create role**.
2. For Trusted Entity Type, select **AWS service** (e.g. select **ECS** or **Lambda**).
3. Search for and check the box next to `BedrockLeastPrivilegePolicy`.
4. Click **Next**, name the role `BedrockAppExecutionRole`, and click **Create role**.


### 2. Ruby

Initiate a client using specific STS role assumption keys:

```ruby
require 'aws-sdk-bedrockruntime'
require 'aws-sdk-sts'

sts_client = Aws::STS::Client.new(region: 'us-east-1')
assumed_role = sts_client.assume_role(
  role_arn: 'arn:aws:iam::123456789012:role/BedrockAppExecutionRole',
  role_session_name: 'bedrock-rails-session'
)

# Initialize bedrock runtime using assumed credentials
client = Aws::BedrockRuntime::Client.new(
  region: 'us-east-1',
  credentials: Aws::Credentials.new(
    assumed_role.credentials.access_key_id,
    assumed_role.credentials.secret_access_key,
    assumed_role.credentials.session_token
  )
)
```

### 3. Ruby on Rails 8

Configure an initializer to verify correct IAM credentials are loaded on application startup:

```ruby
# config/initializers/bedrock.rb
begin
  # Ping the STS service to verify credentials work
  sts = Aws::STS::Client.new(region: ENV.fetch('AWS_REGION', 'us-east-1'))
  identity = sts.get_caller_identity
  Rails.logger.info("Bedrock IAM verified. Caller ARN: #{identity.arn}")
rescue => e
  Rails.logger.warn("Bedrock start warning: AWS credentials missing or invalid (#{e.message})")
end
```

### 4. Next.js 16

Verify role capabilities inside API routes using node SDK STS credential loaders:

```typescript
// app/api/auth-check/route.ts
import { STSClient, GetCallerIdentityCommand } from "@aws-sdk/client-sts";
import { NextResponse } from "next/server";

export async function GET() {
  const client = new STSClient({ region: "us-east-1" });
  try {
    const command = new GetCallerIdentityCommand({});
    const response = await client.send(command);
    return NextResponse.json({
      authenticated: true,
      arn: response.Arn,
      account: response.Account
    });
  } catch (error: any) {
    return NextResponse.json({
      authenticated: false,
      error: error.message || error
    }, { status: 401 });
  }
}
```

### 5. Terraform

Fully provision the application execution role and model execution permissions:

```hcl
resource "aws_iam_role" "app_execution_role" {
  name        = "BedrockAppExecutionRole"
  description = "Execution role for ECS containers or Lambdas querying Bedrock"

  assume_role_policy = jsonencode({
    Version = "2012-10-17"
    Statement = [
      {
        Action    = "sts:AssumeRole"
        Effect    = "Allow"
        Principal = {
          Service = [
            "ecs-tasks.amazonaws.com",
            "lambda.amazonaws.com"
          ]
        }
      }
    ]
  })
}

resource "aws_iam_role_policy" "bedrock_app_policy" {
  name = "BedrockAppAccessPolicy"
  role = aws_iam_role.app_execution_role.id

  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [
      {
        Effect = "Allow"
        Action = [
          "bedrock:Converse",
          "bedrock:ConverseStream"
        ]
        Resource = [
          "arn:aws:bedrock:us-east-1::foundation-model/anthropic.claude-3-5-sonnet-20241022-v2:0",
          "arn:aws:bedrock:us-east-1::foundation-model/amazon.nova-micro-v1:0"
        ]
      }
    ]
  })
}
```

## Interview Questions

**Q: If you need to allow a third-party AWS account to invoke models hosted in your account, how would you design the security architecture?**

*Answer*:

I would configure a cross-account IAM Role:

1. In the hosting account, create an IAM Role (e.g. `BedrockCrossAccountRole`) containing a trust relationship policy that allows the third-party account ID (`sts:AssumeRole`) to assume the role.
2. Attach a least-privilege policy to this role permitting only `bedrock:Converse` on approved Model ARNs.
3. The third-party account can then call `AssumeRole` to fetch temporary credentials, using those to sign requests and communicate securely with our Bedrock endpoints.

## Best Practices & Production Notes

- **Never Use Access Keys in Prod**: Do not save developer `aws_access_key_id` variables in production environments. Always rely on ECS Task Roles, EKS Service Accounts, or Lambda Execution Roles.
- **Model Whitelisting**: Lock down IAM policies to only permit approved model IDs, preventing unauthorized and expensive model tests by applications.
- **Audit Access Regularly**: Run IAM Access Analyzer reports to detect external or open accesses to your Bedrock models, keeping compliance vectors clean.
